# 团队协作约定（Git 分支式，SK0）

> SK0 协作底座，对应设计稿 §9 路线图 D-4。本技能**不内置权限系统**，团队协作
> 完全借助 Git 分支/PR 与本地为唯一事实源的纪律达成。轻量、不引入服务端依赖。

## 一、协作原则

- **本地仓库为唯一事实源**：所有文本契约（project.yaml / shotlist / genspec /
  takes / ledger …）在仓库内可重放、可审计；协作即对这些契约文件的并行编辑与合并。
- **媒体不入仓**：视频/图片等媒体实体走 `media-manifest`，仓库只存清单与契约，
  避免大文件冲突与膨胀（见各子技能媒体纪律）。
- **无内置权限**：谁能改什么由 Git 托管平台（GitHub/GitLab）的分支保护与 Code Review
  决定，本技能脚本不做身份/权限校验。

## 二、分支模型

- `main`：可发布的稳定主线，受保护，仅经 PR 合并。
- `shot/SHOT-xx-<简述>`：单镜工作分支（分镜/生成/选片），按镜并行，互不阻塞。
- `stage/<阶段名>`：阶段性集成分支（如 `stage/storyboard`、`stage/edit`），
  跨多镜协同时使用。
- `fix/<简述>`：QC 命中后的局部重跑/修复分支。

一镜一分支，使抽卡/选片的高频改动局部化，降低跨镜冲突。

## 三、PR 与评审

- 每个分支以 PR 合回 `main` 或集成分支；PR 描述附：涉及镜头、契约变更、
  关键账本/质检结论（成本、命中率、命中失败模式）。
- 评审关注点：契约是否过 `validate`、能力数字是否挂 `verified_at`、半自动项是否
  显式标注"需人工确认"、媒体是否误入仓。
- **不可逆动作（如发布）须经 G9 导演终确认**，PR 合并不等同于发布。

## 四、冲突处理

- 契约文件冲突：优先保留两侧的结构化字段，按设计稿语义人工合并，合并后必须
  重新 `validate`，不得以丢弃一侧改动的方式"图省事"。
- 账本 `ledger/events.jsonl` 为 append-only 单写者流水：冲突时保留双方事件行，
  事件 `event_id` 幂等去重，再 `--rebuild` 重算 `summary`。
- 经验库 `cross-project-lessons.yaml`：同 `project_id`（同 `kind`）覆盖，幂等回写，
  天然抗重复合并。

## 五、与学习回路的衔接

- 结案经验经 `lessons_writeback.py` 回写经验库；观众反馈经 `feedback_intake.py`
  聚合回写（D-7）。二者均以经验文本入活文档，**不进可执行契约的能力数字**。
- 跨项目协作时，新项目冷启动可先读经验库，借鉴历史命中模式与高频负面反馈类别。
